# GWS (Google Workspace CLI) — Tool Reference

`gws` is a Google Workspace CLI covering Gmail, Calendar, Drive, Docs, Sheets, and admin endpoints. Authenticates via OAuth2 against a per-operator Google Cloud OAuth app whose credentials live in Infisical.

This tool layer rides on top of the Infisical foundation (`config/rules/tools/infisical.md`). Don't attempt to set up gws on a machine that hasn't run `bootstrap-infisical.sh`.

## Access Method

CLI binary: `gws`, installed by `setup-gws.sh` via `brew install googleworkspace-cli` (preferred) or `npm install -g @googleworkspace/cli` (fallback). Both methods come from the official repo at https://github.com/googleworkspace/cli — do NOT `brew install gws`, which installs an unrelated git-workspaces tool of the same name.

A shell wrapper at `config/shell/gws-env.sh` makes `gws auth login` pull the OAuth-app `client_id`/`client_secret` from Infisical at the moment of login — no env vars in your shell, no plaintext on disk. Source from `~/.zshrc`:

```bash
source /path/to/Workdesk-OS/config/shell/gws-env.sh
```

The wrapper is a pass-through for every command except `gws auth login`, which it intercepts to inject the OAuth-app credentials.

## Setup

Run once per machine, after `bootstrap-infisical.sh`:

```bash
bash config/scripts/setup-gws.sh
```

That script:
1. Verifies the Infisical foundation is in place.
2. Installs gws if missing (`brew install googleworkspace-cli`, falling back to `npm install -g @googleworkspace/cli`).
3. Renders the 4 gws agent-templates from `.tmpl.in` → `.tmpl` with your operator-config values.
4. Appends the gws templates block to `config/infisical/agent.yaml`.
5. Migrates `~/Library/Application Support/gws` to a symlink pointing at `/Volumes/wd-ramdisk/gws/`. Existing auth state survives the move via a backup at `gws.bak.<timestamp>`.
6. **If you were already authenticated locally** — pushes your auth state to Infisical, *then* reloads the Infisical Agent. Race-safe ordering: Infisical has values before the agent's first render pass.
7. **If you weren't yet authenticated** — stops at step 5 without reloading the agent. You then run `gws auth login` and re-run this script; on the second pass it detects the new auth state and finishes the wire-up.

Why the conditional in step 6: if the agent reloads before Infisical has your state, the first render pass finds nothing in Infisical, writes empty files to the ramdisk, and the post-render hook decodes empty into empty — clobbering your just-migrated auth state. Pushing before reloading eliminates the window.

If the script reloaded the agent for you, the only remaining tasks are verification and (optionally) sourcing the shell wrapper in `~/.zshrc` so future `gws auth login` calls pull OAuth-app creds from Infisical automatically.

## Where state lives

`gws` keeps OAuth state in `~/Library/Application Support/gws/`. After setup, that's a symlink to `/Volumes/wd-ramdisk/gws/` so nothing persists to the SSD. Files in that directory:

| File | What | Source of truth |
|---|---|---|
| `client_secret.json` | OAuth app creds (client_id/secret/project_id) | Infisical: rendered by the Agent from the three `PERSONAL_GOOGLE_WORKSPACE_*` keys |
| `.encryption_key` | Per-install random key that wraps the refresh token | Infisical: `PERSONAL_GWS_ENCRYPTION_KEY` |
| `credentials.<b64-email>.enc` | Encrypted OAuth refresh token | Infisical: `PERSONAL_GWS_CREDENTIALS_<KEY_SUFFIX>_ENC_B64` (stored base64; decoded by post-render hook) |
| `accounts.json` | Registered accounts list | Infisical: `PERSONAL_GWS_ACCOUNTS_JSON` |
| `token_cache.<b64-email>.json` | Short-lived access token (~1h) | Regenerated by `gws` from the refresh token — not round-tripped |

The Infisical Agent (configured in `config/infisical/agent.yaml`) renders the durable files into the ramdisk at boot. `token_cache.*.json` is recreated on the fly by `gws` itself — ephemeral, no need to persist.

## Common Commands

| Command | What it does | Example |
|---|---|---|
| `gws auth status` | Show current auth state | `gws auth status` |
| `gws auth login --account EMAIL` | OAuth login flow (opens browser) | `gws auth login --account you@example.com` |
| `gws <svc> <res> <method> --params '...'` | Generic Google API call | `gws calendar events list --params '{"calendarId":"primary"}'` |
| `gws schema <svc>.<res>.<method>` | Inspect API surface | `gws schema gmail.users.messages.list` |

## When to re-run `gws-push-tokens-to-infisical.sh`

The script at `config/scripts/gws-push-tokens-to-infisical.sh` mirrors the local refresh-token state back into Infisical. Re-run when:

- You re-authenticate (`gws auth login`) — the refresh token changes.
- You add a new Google account to gws — `accounts.json` and `credentials.<new-email>.enc` need pushing. (Multi-account support beyond the primary `email` field in operator-profile is not yet wired through Infisical; one account per operator for now.)
- You rotate the encryption key (rare; happens if you `gws auth logout --all` and re-login).

If you skip the push and reboot, gws will think it has never been authenticated and prompt for the browser OAuth flow again.

## Trust boundary

`PERSONAL_GWS_ENCRYPTION_KEY` + `PERSONAL_GWS_CREDENTIALS_<KEY_SUFFIX>_ENC_B64` together are equivalent to plaintext access to your Gmail/Drive/Calendar. Anyone with read access to these two Infisical keys can impersonate your Workspace identity. The Universal Auth machine identity used by the Agent should be **Viewer (read-only)** and scoped to your personal project only — never granted write.

## Known Limitations

- **Ramdisk is volatile.** A power loss or kernel panic kills the access-token cache. Next call rehydrates from the refresh token via the agent — adds ~1s latency, no auth prompt.
- **Reboot path depends on Infisical Cloud being reachable.** If Infisical is down at login, the agent fails to render and gws has nothing to read. Fallback: backup at `~/Library/Application Support/gws.bak.*` (created automatically by `gws-migrate-to-ramdisk.sh`) until you've confirmed the new flow works.
- **Multi-account gws not yet wired through Infisical.** Templates cover the primary `email` from operator-profile only. Multi-account requires extending `agent-gws.yaml.snippet.tmpl` with additional credentials-file template entries and adding new `PERSONAL_GWS_CREDENTIALS_<SUFFIX>_ENC_B64` keys.

## Common Mistakes

- **Re-authenticating without re-running `gws-push-tokens-to-infisical.sh`.** The ramdisk gets the new token, but Infisical still has the old one. Next reboot reverts to the stale token, which Google will reject — forcing a re-auth. Always pair `gws auth login` with the push script.
- **Setting `GOOGLE_WORKSPACE_CLI_CLIENT_ID` permanently in `~/.zshrc` instead of via the wrapper.** Puts the value in plaintext in your shell history and dotfiles. Use the wrapper at `config/shell/gws-env.sh`.
- **Treating `~/Library/Application Support/gws/` as a real directory after migration.** It's a symlink. Delete the symlink and gws state is gone; the ramdisk dir is the real location.

## Detection clause

Surface proactively when:

- The operator references something only Google can answer (calendar event, email, Drive doc) and `gws auth status` would resolve it — propose using gws instead of guessing.
- The operator says they re-authenticated gws but didn't push tokens — remind them to run `gws-push-tokens-to-infisical.sh` before the next reboot.
- A new Google account needs adding — note that multi-account isn't wired yet and explain the extension path (new template entry + new `PERSONAL_GWS_CREDENTIALS_<NEW_SUFFIX>_ENC_B64` key).
- `gws auth status` shows "not authenticated" right after a reboot — check that the Infisical Agent rendered the ramdisk (look at `/Volumes/wd-ramdisk/gws/`), and that the agent log shows successful authentication.
